VARIANCE THRESHOLDS

Document allowable variances and how baselines will be managed.

Scope Variance Threshold

  • Minor UI tweaks and sub-features that do not change main user flows.
  • Performance: adding a transaction in < 100 ms on mid-range device.
  • Rendering smoothness: target ≥ 60 FPS for main screens.

Scope Baseline Management

Controlled via Scope Baseline (approved requirements and WBS). Minor changes logged to backlog without CR. Medium/major changes require Change Request with impact analysis (Schedule/Cost/Quality) and approval before implementation. Update the requirements traceability matrix post-approval.

Schedule Variance Threshold

  • Acceptable: ≤ 10% task slip, no critical path impact, ≤ 50% float used.
  • Warning: 10–20% slip or > 50% float used.
  • Unacceptable: > 20% slip or critical path delay > 3 days/1 week.

Schedule Baseline Management

Weekly tracking (burndown, % complete). On warning: re-prioritize, trim minor tech debt. If unacceptable: raise CR or de-scope low-priority features; add overtime if permitted. Update baseline upon CR approval.

Cost Variance Threshold

  • 0–5% effort variance acceptable.
  • 6–10% triggers investigation and corrective actions.
  • > 10% triggers urgent corrective action and possible re-baselining.

Cost Baseline Management

Simple EVM tracking: PV/EV/AC in hours; report CV and CPI weekly; take actions per thresholds and re-baseline upon approved changes.

BASELINES
  • Scope Baseline: requirements, WBS, traceability matrix
  • Schedule Baseline: Gantt (W1–W8) with milestones (M1–M4)
  • Cost Baseline: effort estimates by WBS, 10% contingency
  • Quality Baseline: acceptance criteria and key quality metrics
